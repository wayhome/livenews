GIMP 3.0

作者: wicket | 评分: 390 | 评论数: 28 | 发布时间: 2025-03-17 23:25
文章摘要

GIMP 3.0正式发布，经过七年的开发，带来了多项重大改进。新版本支持大多数常用滤镜的非破坏性编辑，并实时预览更改。文件交换能力增强，支持更多格式如BC7和DDS文件，PSD导出也得到改进。绘图工具可以自动扩展图层，文本编辑功能更加专业，支持样式、轮廓、阴影等效果。图层管理更便捷，支持多项同时操作。色彩管理进一步优化，采用了新的GTK3工具包，并更新了Wilber和GIMP的标志。更多详细信息可查阅发布说明和 changelog，用户可从官网下载体验。同时，GIMP 3.0还发布了相关组件和手册。社区捐赠可以支持GIMP的持续发展。

主要讨论点：GIMP 3.0版本的新功能、用户体验（UI/UX）以及非破坏性编辑的引入 不同观点： • **geenat**：肯定了GIMP 3.0在简单桌面出版和YouTube缩略图制作中的潜力，但指出了多个UI/UX问题。包括：操作摩擦过大（如通过"Tool->GEGL Operation..."进行常见操作）、命名不直观（如阴影和发光效果隐藏在"GEGL Styles"中）、工具切换不便（如"Move Tool"没有快捷切换到其他工具）、图层样式复制/粘贴功能缺失，以及一个具体的BUG（图层常常裁剪GEGL Glow效果）。 • **idle_zealot**：认为GIMP 3.0在文本编辑上的改进（如样式、轮廓、阴影、斜面等）使得高质量文本制作更加容易，是工作流的重大改进，甚至可以媲美专有编辑软件。 • **tasuki**：对“高质量文本”功能持怀疑态度，认为这些功能只是“廉价的花招”，并指出非破坏性编辑的引入是积极的改变。同时提到自己已转向DarkTable进行照片编辑，认为GIMP的目标并非照片编辑。 • **Carrok**：长期使用GIMP，对UI/UX有感情，尽管存在不足，但仍认为GIMP是顶级免费软件。 • **ryukoposting**：对非破坏性编辑的引入感到兴奋，认为这是GIMP的巨大进步，但个人更偏好Rawtherapee，可能不会回归GIMP。 • **esskay**：批评发布公告中缺乏截图。 • **ChrisArchitect**：指出发布链接可能有误，提供了修正链接。 • **anymouse123456**：祝贺并感谢开发团队的辛勤工作，作为轻度用户，对GIMP多年来的小UX细节和性能改进表示赞赏。 • **xtagon**：对非破坏性编辑的引入表示欢迎。 • **rascul**：关注GTK3的更新，并质疑Linux发行版是否会放弃GTK2。 • **jayzalowitz**：感叹GIMP项目的历史悠久，对新版本的发布表示赞赏。 • **dskhatri**：长期使用GIMP，特别提到BIMP插件在批量图像处理中的实用性。 • **bhouston**：认为GIMP在2D和3D编辑中取代了专有工具，并列出GIMP 4的愿望清单，包括AI工具的集成（如生成式填充、AI超分辨率、Meta的“分割任何东西”功能、内置模型上下文协议支持等）。 • **synergy20**：认为GIMP对普通用户和简单需求来说过于复杂，希望有一个简化版本，类似于手机照片应用的一键模式。 • **jasonjamerson**：曾打开过GIMP，但对UI更新不满意，表示除非UI更新能与Blender的大更新相媲美，否则不会再次使用。 补充讨论： • 争议焦点：GIMP 3.0在文本编辑功能上的改进是否能达到“高质量”标准存在不同看法（idle_zealot vs tasuki）。 • 用户对UI/UX的改进需求较为集中，特别是在工具切换、图层样式操作和功能发现性方面。 • 非破坏性编辑的引入得到了广泛认可，但用户对其具体应用和效果有不同期待。 • 一些用户对AI工具的集成表现出浓厚兴趣，并提出了具体功能建议。

Past and Present Futures of User Interface Design

作者: sedboyz | 评分: 33 | 评论数: 5 | 发布时间: 2025-03-17 23:35
文章摘要

这篇文章回顾了用户界面设计的历史和未来愿景。1968年，Douglas Engelbart展示了首个计算机系统NLS，引入了视频会议、协作文本编辑、超文本等功能，这一演示被称为“所有演示之母”。随后，Xerox Alto成为首个具有图形用户界面（GUI）的电脑，启发了后来的个人电脑设计。文章还提到Sun Microsystems的Starfire电脑广告，展示了触摸屏、视频会议等未来概念。作者Amelia Wattenberger提出未来界面应增加触觉“摩擦”，赞扬多模态交互，并建议在输入设备和反馈选项上进行多样化。文章总结了许多过去的创新实际上已经出现在现代界面中，而未来界面可能结合触控、语音和更大的显示屏。

评论摘要
主要讨论点：不同输入方式（键盘、语音、触屏、物理按钮等）的优缺点及未来发展 不同观点： • [gyomu] 认为键盘作为输入方式有其持久的存在价值，尽管不断有人尝试用触屏、语音输入等方式取代它。他基于在设计研究实验室的工作经验指出，尽管新输入方式在理论上可能有优势，但在实际使用中往往不如键盘有效。他主张设计师应更倾向于描述性（descriptivist）而非规定性（prescriptivist），即观察用户如何实际使用界面，而不是试图改变他们的习惯。他同时强调原型设计和用户测试的重要性，批评一些设计师过于注重演示效果而非实际可用性。 • [tony-allan] 偏好使用鼠标而非键盘快捷键，因为难以记住各种环境下的快捷键。他提出了一种更为触觉化的界面设计，包括物理旋钮、触屏和宏键盘等，认为这些设计可以提升操作效率。他提到专业视频和音频工程师已经使用类似技术，并表示自己有兴趣尝试。 • [furyofantares] 认为对语音控制的讨论过于简单化，没有深入探讨自然语言作为界面输入的潜力。他质疑为什么没有讨论自然语言是否可以作为当前文本输入的有力替代或补充。 • [DidYaWipe] 认为原文对语音控制的批评过于表面化，且没有提及曾经被高度期待但最终失败的“少数派报告”式界面设计，暗示对未来界面设计的讨论应考虑历史上类似失败案例的教训。 补充讨论： • 对新输入方式的实际可用性和用户习惯的重视：[gyomu] 和 [tony-allan] 都提到新输入方式在实际使用中的不足，强调用户习惯和操作效率。 • 对自然语言界面潜力的忽视：[furyofantares] 指出原文未深入探讨自然语言作为界面的可能性，这可能是一个值得进一步研究的领域。 • 对未来界面设计失败案例的反思：[DidYaWipe] 提醒注意历史上如“少数派报告”式界面设计的失败，强调对未来设计的现实考量。 争议焦点： • 新输入方式（如语音控制、触屏）是否能够真正取代键盘，成为主流输入机制。 • 界面设计中，如何平衡创新与实际可用性，避免重蹈历史上失败设计的覆辙。

Archival Storage

作者: rbanffy | 评分: 212 | 评论数: 22 | 发布时间: 2025-03-17 18:36
文章摘要

本文总结了在伯克利信息学院信息访问研讨会上的一次演讲，主题是关于**档案存储**。演讲者强调不要轻信所谓“准永久介质能解决档案数据存储问题”的炒作。他详细介绍了自己的备份策略，包括使用Raspberry Pi、DVD-R等设备进行定期备份，并强调**备份**与**存档**是不同的问题。备份的目的是应对灾难恢复，数据寿命与介质寿命无关；而存档则是为了降低成本，接受更长的访问延迟。演讲者还提到目前消费级光盘介质虽然有效，但市场正在消亡，因为消费者对长期存档兴趣不大。最后，讨论了超长期数字存档的挑战，如10,000年的档案设计目标。

评论摘要
主要讨论点：数据存储的长期保存问题及不同解决方案 不同观点： • **entrepy123** 认为，尽管技术进步显著，但普通数据的保存依然面临挑战，通常只能维持约5年。他指出，长期保存数据需要大量精力和成本，例如使用JBOD、RAID、NAS或M-Disc等方法，或者依赖云存储。 • **alnwlsn** 提出，传统的物理媒介如雕刻、纸张、摄影胶片等可能比数字媒介更持久。他讨论了通过微缩胶片进行长期保存的可能性，并指出目前多数结果是将物理媒介数字化，而非保存原始物理媒介。 • **PaulHoule** 认为，企业级和个人级的成本计算不同。他质疑LTO磁带的可靠性，基于其过去的糟糕经验，并倾向于使用大规模生产的硬盘。 • **jewel** 建议使用云存储时启用Object Lock功能，以提高数据安全性，并分享了使用restic和B2进行备份的经验。 • **nntwozz** 采用3-2-1备份策略，使用SSD和Blu-ray光盘存储数据，以防范地磁风暴等自然灾害。 • **hn_throwaway_99** 指出，设计为失败并购买最便宜的组件对大型企业有意义，但对于个人或小公司不适用。他使用廉价的外置硬盘和M-DISC Blu-rays进行数据存储。 补充讨论： • **rtkwe** 认为磁带存储因其昂贵的驱动器而难以进入普通市场。 • **markhahn** 对LOCKSS系统提出疑问，关注其数据检查和验证机制。 • **lizknope** 分享了其自1991年以来的文件存储经验，采用3-2-1备份策略并定期校验文件。 • **UltraSane** 强调LTO磁带因其设计目的和低错误率而适合长期存储，但需同时保存兼容的驱动器。 • **CartwheelLinux** 提到中国有自己的光存储实现，一些光盘可存储高达1TB的数据，但相关资料多为中文。 • **mburns** 建议使用M-Disc解决光盘退化问题。 • **xhrpost** 引用了关于50%磁带备份失败的文章，质疑磁带备份的可靠性。 • **lvl155** 提出在太空（如月球）存储数据的可能性，这是一个较为超前的话题。 争议焦点： • 不同存储介质的可靠性和成本效益，例如LTO磁带与硬盘的对比。 • 个人与企业级数据存储策略的差异，以及如何在成本和可靠性之间取得平衡。 总结：讨论围绕数据存储的长期保存方案展开，涵盖了物理媒介与数字媒介的对比、不同存储技术的可靠性和成本、以及具体的备份策略和工具选择。

Deep Learning Is Not So Mysterious or Different

作者: wuubuu | 评分: 290 | 评论数: 12 | 发布时间: 2025-03-17 16:47
文章摘要

这篇文章由Andrew Gordon Wilson撰写，探讨了深度学习是否如通常认为的那样神秘和与众不同。文章指出，深度神经网络所展现的诸如良性过拟合、双下降现象和过参数化成功等反常泛化行为，并非神经网络独有，也并非特别神秘。这些现象可以通过已有的泛化理论框架（如PAC-Bayes和可数假设界限）直观理解和严格描述。文章提出“软归纳偏好”作为解释这些现象的关键原则，即不限制假设空间，而是偏好简单且与数据一致的解决方案。尽管深度学习在表征学习和模式连接等方面有独特之处，但总体上它并不比其他模型类别更神秘或不同。

评论摘要
主要讨论点：机器学习（ML）和深度学习（DL）的理论基础、资源推荐及其实际工作原理，包括对深度学习模型如GPT的讨论。 不同观点： • [rottc0dd] 推荐了斯坦福大学的“Probability for Computer Scientists”课程作为学习ML理论基础的资源，并认为Andrew Ng的课程对数学基础要求较高。同时推荐3b1b的视频作为深度学习的视觉化入门。 • [cgdl] 不同意PAC-Bayes或VC理论的解释力，认为算法稳定性提供了更有说服力的解释，并引用了相关的学术文章。 • [TechDebtDevin] 推荐了Josh Starmer的《StatQuest Illustrated Guide to Machine Learning》一书，认为其能以简洁易懂的方式解释复杂的ML概念，适合初学者和专家。 • [getnormality] 质疑深度学习中复杂性惩罚的直接性，提出深度学习是否如传统方法那样通过优化成本函数来实现“软偏好简单解决方案”的问题。 • [d_burfoot] 认为DNNs没有特别的泛化能力，其表现依赖于巨大的数据集和模型复杂性，并通过LLMs的训练范式说明其“魔法”来源于数据的复用性和模型复杂性。 • [inciampati] 引用了一篇关于RNNs的论文，说明深度网络在某些情况下（如长依赖学习）的必要性。 • [buffalobuffalo] 认为理解通用近似定理的证明有助于更好地理解神经网络。 补充讨论： • [YesBox] 提出了一种基于词频和距离的简单文本预测算法，并询问其与GPT-2的相似度。 • [totetsu] 提出了AI监管的界限问题。 • [talles] 认为人工神经元本质上是线性回归加上激活函数，通过网络结构实现复杂功能。 • [EncomLab] 反对将软件视为“神秘黑盒”的观点，认为这种说法是广告术语，强调软件运行状态的可确定性。 争议焦点： • 深度学习模型的复杂性惩罚机制是否直接以及其与传统方法的比较。 • 关于PAC-Bayes和VC理论在解释ML泛化能力上的有效性争议。 • 对深度学习是否神秘以及是否应被视为“黑盒”的讨论。 这些讨论反映了ML和DL领域中不同学习资源、理论解释和实际应用之间的多样性和复杂性。

文章摘要

HTTP/3 is everywhere but nowhere

作者: doener | 评分: 312 | 评论数: 50 | 发布时间: 2025-03-14 07:02
文章摘要

文章《HTTP/3无处不在却又 nowhere》由Tim Perry于2025年3月撰写，讨论了HTTP/3及其底层协议QUIC的发展现状。尽管HTTP/3和QUIC自2016年和2013年分别开始开发和引入，并已在95%的浏览器中得到支持，且在32%的Cloudflare HTTP请求中使用，但其实际应用仍非常有限。大多数主流编程语言的标准库尚未支持HTTP/3，包括Node.js、Go、Rust、Python和Ruby。即使是流行的服务器软件如Nginx和Apache，也仅提供实验性支持或完全没有支持计划。 文章指出，虽然HTTP/3在浏览器和CDN中取得进展，但在其他客户端和服务器实现中仍未普及。HTTP/3的优势不仅限于浏览器和CDN，对于移动应用、API服务器和物联网设备等同样重要。HTTP/3提供了多路复用、头部压缩等优势，可以有效减少延迟和网络开销，尤其在移动网络和不稳定网络环境下更为显著。文章最后强调，尽管HTTP/3的推广刚刚开始，但其潜在价值不容忽视。

评论摘要
主要讨论点：HTTP/3 技术的采用情况、支持现状及其在不同场景下的实用性 不同观点： • djha-skin 认为，HTTP/3 在数据中心场景中并不必要，因为 HTTP/1.1 已经足够且更容易管理和调试。HTTP/3 更适合移动网络中的不稳定连接，而在数据中心静态、低延迟的连接环境下，HTTP/3 引入了不必要的复杂性和开销。 • CharlieDigital 指出，主流编程语言的标准库尚未包含 HTTP/3 和 QUIC 支持，且现有的开源实现可能还不够成熟，难以应用于生产环境。对大多数开发团队来说，HTTP/3 并不是优先考虑的优化点。 • ralfaroo 担心 HTTP/3 的广泛采用会增加代码的安全维护成本，并认为在大多数情况下，几毫秒的延迟差异对用户体验影响不大。 • fresh_broccoli 将 QUIC 缓慢采用的原因归咎于 OpenSSL 早期未提供必要的 API 支持，但指出 OpenSSL 3.5 即将解决这一问题，不过其推送式 API 与现有实现有所不同，仍需观察其对生态系统的影响。 • xg15 提出，HTTP/2 和 HTTP/3 更像是传输层和加密层的协议，开发者主要仍处于 "HTTP 1.1" 的世界中，这些底层协议的变化对应用开发者影响有限。 • jsheard 批评 NGINX 在 HTTP/3 支持方面的进展缓慢，并推荐使用 Python 的 niquests 库作为支持 HTTP/3 的替代方案。 • jillesvangurp 认为，使用公有云提供商负载均衡器的服务已经默认支持 HTTP/3，但自建服务器则需要额外工作，且 HTTP/3 在服务器间的 API 调用中优势不大。 • artyom 指出，HTTP/3 的开源项目和社区支持进展缓慢，与 IPv6 类似，可能需要很长时间才能广泛采用。 • jiehong 提到，Caddy 已经支持 HTTP/3，可以用作反向代理，但大多数 HTTP 客户端库仍缺乏支持。 • raggi 质疑 HTTP/3 对未采用它的用户的实际意义，认为其主要适用于大规模托管服务和视频流场景，而这些场景的用户已经采用了该技术。 • huijzer 反问 Caddy 是否已经是一个具备完善 HTTP/3 支持的工具，并质疑原文所述的 "rollout has barely even started"。 补充讨论： • 一些评论提到了 HTTP/3 在不同操作系统和网络环境下的实际性能问题，特别是 UDP 缓冲区设置可能影响吞吐量（raggi）。 • 讨论中多次提到 OpenSSL 的支持问题及其对 QUIC 和 HTTP/3 推广的影响（fresh_broccoli, jillesvangurp）。 • 关于 HTTP/3 的实际应用场景，评论中存在争议，一部分人认为其在数据中心和服务器间通信中的优势有限（djha-skin, jillesvangurp），而另一部分人认为其在移动网络和浏览器端有显著优势（jillesvangurp, CharlieDigital）。 争议焦点： • HTTP/3 是否在数据中心和服务器间通信中具有显著优势，以及是否值得广泛采用。
