What is vibe coding? How creators are building software with no coding knowledge

作者: Bocajmai | 评分: 18 | 评论数: 13 | 发布时间: 2025-03-01 11:57
文章摘要

"Vibe coding"是一种利用AI构建软件的新方法，由特斯拉和OpenAI前工程师Andrej Karpathy提出。它让即使没有技术背景的创作者也能通过自然语言描述来创建功能性应用程序，而无需直接编写代码。与传统编程需要数周或数月的学习和开发不同，vibe coding只需几分钟学习，几小时内即可构建出简单工具。这种方法通过AI自动生成和修复代码，大大降低了开发门槛，特别适合需要快速实现创意的创作者。它让创作者无需掌握复杂编程技能，即可为受众定制工具并快速测试想法，从而改变内容创作和社区管理的方式。

评论摘要
主要讨论点：围绕“vibe编码”（即依赖AI生成代码的编程方式）的可行性、实用性及其潜在问题展开的讨论。 不同观点： • **支持vibe编码**： - [UrineSqueegee] 认为vibe编码对业余爱好者和小型项目非常实用，能节省雇佣专业人员的成本。他相信随着技术进步，AI生成的代码会逐渐变得更可维护和安全。 - [yapyap] 认为vibe编码是“基于感觉”的编程方式，类似于无计划直接开始项目的KANBAN方法，尽管更依赖AI的“感觉”。 • **质疑vibe编码**： - [nkrisc] 认为vibe编码和业余爱好者长期以来的做法没有本质区别，担心AI可能会涉及个人敏感信息或收费问题。 - [iammrpayments] 指出，虽然非程序员可以通过AI创建功能性软件，但这与使用WordPress或Shopify建电商网站类似，仍然存在很多限制。 - [rvz] 强烈反对vibe编码，称之为“无稽之谈”，认为缺乏软件维护知识的人使用这种方式会带来问题，将其比作“无限猴子定理”，并预测这种热潮最终会像1999-2000年的科技泡沫一样破灭。 • **中立或调侃态度**： - [someothherguyy] 认为所谓的vibe编码与过去使用代码片段或参考书编写代码没有太大区别，只是工具变了。 - [rakoo] 调侃说“写完即弃”的程序时代回来了，Perl语言又要变得相关了。 - [zeristor] 提到调试快速开发软件（RAD）已经够痛苦，暗示vibe编码可能会带来更多调试难题。 - [59327ZAG] 批评AI的宣传和炒作，并引用政治人物的言论，暗示对AI的盲目信任可能导致问题。 补充讨论： - 争议焦点在于vibe编码是否真正有效以及其未来发展潜力。支持者强调其便捷性和潜力，而反对者则担心代码质量和可维护性问题。 - 讨论中多次提到软件工程的严肃性和安全性问题，特别是对于关键系统（如火箭、国防技术）的应用，暗示vibe编码在这些领域不适用。 - 调侃和讽刺的评论反映了一些人对AI生成代码的怀疑态度以及对技术炒作的担忧。

When eBPF pt_regs reads return garbage on the latest Linux kernels, blame Fred

作者: tanelpoder | 评分: 80 | 评论数: 2 | 发布时间: 2025-03-01 01:37
文章摘要

文章主要介绍了在Linux内核6.9版本之后，x86_64架构上引入了一个新功能CONFIG_X86_FRED，该功能会在任务的内核栈起始处增加16字节的填充。这导致使用eBPF工具（如xcapture-next v3）读取其他线程的系统调用信息时出现错误结果，因为这些工具直接读取内核内存中的`task_struct`信息。具体表现为，所有用户空间线程的系统调用都被错误报告为`getsockname`，而实际上是由于新内核中栈结构的变化导致读取到了错误的值（如错误码51）。 问题的根本原因是FRED功能引入的栈区域变化，它为内核栈增加了16字节的填充，导致`pt_regs`结构的偏移发生了变化。文章建议在处理内核栈和`pt_regs`查找代码时考虑这额外的16字节。FRED功能本质上是Intel CPU在系统调用时进一步优化权限切换的一种机制，类似于从`int 0x80`到`syscall`/`sysenter`的演变。 解决方法是调整eBPF代码以适应新内核的变化，使用如`BPF_CORE_READ_INTO`宏来正确处理内存读取。

评论摘要
主要讨论点：Linux 内核中的 eBPF 程序与内核结构变化的关系 不同观点： • [blahgeek] 认为这并不是 Linux 内核的 bug。他指出，内部结构体（struct）的布局从来都不是用户空间 API 的一部分，因此其变化是正常的。eBPF 程序应当根据即将执行的内核配置进行编译，就像任何内核模块一样。 • [awsanswers] 则对这一话题表示兴趣，并提到这是在用户空间的边缘进行操作，暗示这种情况涉及用户空间与内核空间的交互，存在一定的复杂性和挑战。 补充讨论： • [blahgeek] 强调了内核模块和 eBPF 程序的编译需要与特定内核配置相匹配，这意味着开发者需要考虑内核内部结构的变化。 • [awsanswers] 没有直接反驳 [blahgeek] 的观点，但对在用户空间边缘操作表现出关注，可能意味着对该操作的稳定性和可维护性存在一定担忧。 • 争议的焦点在于内部结构体变化是否应被视为问题。一方认为这是预期行为，另一方则关注其在特定操作环境下的潜在影响。
