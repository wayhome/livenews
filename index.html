New speculative attacks on Apple CPUs

作者: cylo | 评分: 844 | 评论数: 34 | 发布时间: 2025-01-28 18:31
文章摘要

本文介绍了SLAP和FLOP两种针对Apple Silicon处理器的推测执行攻击。SLAP利用了M2/A15及更高版本Apple CPU中的负载地址预测器（LAP），通过错误的地址猜测获取不应访问的数据，从而实现攻击，例如远程攻击者可获取Safari浏览器中的电子邮件内容和浏览行为。FLOP则针对M3/A17及更新版本CPU的负载值预测器（LVP），通过错误的数据预测绕过内存安全检查，泄露敏感信息，如位置历史、日历事件和信用卡信息。这些攻击突破了网页隔离保护，展示了硬件漏洞的严重安全风险。受影响设备包括2021年9月以来的Apple设备。防御措施尚在研究中。

评论摘要
主要讨论点：针对SLAP漏洞及其在不同浏览器和硬件上的影响，以及可能的缓解措施和业界反应。 不同观点： • omcnoe认为Safari浏览器在调用`window.open`时没有将新窗口隔离在单独的进程中，导致漏洞的有效性。他强调了进程隔离对漏洞利用的重要性。 • tptacek对M3处理器上LVP（负载值预测器）的细节进行了评论，指出零值加载的特殊性可能是一种内存安全对策。 • bawolff提到通过缓存放大原语区分缓存命中和未命中的技术，质疑之前对高精度定时器和SharedArrayBuffer的禁用措施是否有效。 • saagarjha讨论了Chrome的Site Isolation并非完美缓解措施，指出子域名可能被合并到同一进程中，并观察到iPhone 13和13 mini在LAP（负载地址预测器）上的差异。 • yoshicoder分享了其教授Daniel Genkin在SLAP攻击上的工作背景，并表达了对安全性的担忧。 补充讨论： • tptacek提到RUB（鲁尔大学波鸿分校）在高阶攻击性安全研究上的地位。 • daneel_w关注苹果最近的安全补丁是否解决了这些漏洞。 • anotherhue质疑苹果声明是否只是公关，并提供相关链接供讨论。 • hk1337对私密浏览模式下是否受此漏洞影响表示好奇。 • phendrenad2讨论了推测执行的不安全性，并质疑CPU架构的未来方向。 • junek询问在漏洞修补前可以采取的缓解措施。 • wswin建议禁用不信任网站上的JavaScript，认为这是当前有效的防护手段。 争议焦点： • 各浏览器和硬件厂商的缓解措施是否足够有效，特别是Safari的进程隔离问题和Chrome的Site Isolation。 • 苹果公司对漏洞的官方声明是否只是公关回应，以及其潜在风险是否被低估。 其他值得注意的讨论点： • 研究项目的资金来源及其未来资助可能性。 • 普通用户在漏洞修补前可以采取的防护措施，如禁用JavaScript或使用私密浏览模式。

Composable SQL

作者: earnestinger | 评分: 156 | 评论数: 24 | 发布时间: 2025-01-26 09:08
文章摘要

本文讨论了SQL在测试和重用业务逻辑方面的不足，并提出了通过引入具有静态类型接口的可组合查询片段来改善SQL的方案。 **主要问题：** 1. **测试困难**：由于SQL查询依赖整个数据表的结构，测试时需要填充所有列和相关表的数据，即使某些数据对查询结果没有影响，导致测试数据准备工作繁琐且测试覆盖不均。 2. **业务逻辑重用性差**：在完全规范化的数据库中，业务逻辑往往在读取时实现，涉及动态计算和复杂查询，难以重用和维护。 **解决方案：** 引入**函子（Functors）**以创建可组合的查询片段，提供静态类型接口，解决上述问题。函子可以： - **简化测试**：通过隔离依赖，减少不必要的数据填充。 - **重用业务逻辑**：将动态计算和复杂查询逻辑模块化，方便复用和维护。 该方案旨在提高SQL的可测试性和业务逻辑的重用性，从而优化数据库操作的效率和可靠性。

评论摘要

Using uv as your shebang line

文章摘要

本文介绍如何使用`uv`工具通过 shebang 行直接在命令行运行 Python 脚本。作者在 `~/bin/` 目录下创建了许多自动化任务的脚本，并通过 `uv run --script` 命令及 shebang 行 `#!/usr/bin/env -S uv run --script` 实现脚本的直接执行。`uv` 会忽略 shebang 行，将文件作为普通 Python 文件运行。确保脚本具有可执行权限后（通过 `chmod a+x {filename}`），就可以方便地在命令行运行依赖自动处理的 Python 脚本。
