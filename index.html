Sign in as anyone: Bypassing SAML SSO authentication with parser differentials

作者: campuscodi | 评分: 207 | 评论数: 16 | 发布时间: 2025-03-15 19:06
文章摘要

文章主要介绍了在ruby-saml库（用于SAML单点登录）中发现的关键身份验证绕过漏洞（CVE-2025-25291和CVE-2025-25292）。攻击者只需拥有目标组织用于验证SAML响应或声明的有效签名，即可伪造SAML声明，从而登录为任意用户，导致账户接管风险。用户应更新至ruby-saml 1.18.0版。GitHub虽不再使用该库，但在重新评估其使用可能性时，通过漏洞奖励计划发现了这一问题。研究发现，ruby-saml在签名验证过程中使用了两种不同的XML解析器（REXML和Nokogiri），可能导致解析差异，进而被利用进行身份验证绕过。文章详细描述了漏洞发现和验证的四个阶段，最终确认了该漏洞的严重安全性影响。

评论摘要
主要讨论点：GitHub SAML 实现及其相关安全问题 不同观点： • asmor 认为 GitHub 的 SAML 实现存在安全漏洞，尤其是在组织成员信息泄露方面。他指出，授权应用程序后，即使通过 SAML 会话控制，组织成员信息仍可能被读取。他提到几个厂商的不同反应，并认为报告安全漏洞没有得到足够的重视。 • Diggsey 批评 SAML 规范本身基于 XML 签名和 XML 规范，这些标准非常复杂且容易出错，难以实现。 • tptacek 强烈反对使用 SAML 和 XML-DSIG，认为 OAuth 是更好的选择。他认为 SAML 是非常危险的协议，可能还会出现更多漏洞。 • kayodelycaon 特别指出 REXML 解析器的缺陷，认为使用像 Nokogiri 这样更严格的解析器会更安全。 • noleary 对 ahacker1 在 SAML 实现安全方面的工作表示赞赏，并提到 WorkOS 对其工作的认可。 • derektank 提到 GitLab 也修复了相关的漏洞，提供了修复链接。 • RainyDayTmrw 引用了 Latacora 的文章，讨论了如何正确签署 JSON 对象，暗示 SAML 的复杂性容易导致问题。 • wcoenen 认为应在正确位置查找签名，而不是使用过于通用的 XPath，这可能是避免漏洞的一种方式。 • bawolff 指出文章中未提及具体的解析器差异，这使得漏洞描述不够完整。 • fsfsd43535 认为 REXML 和 Nokogiri 的实现差异导致了安全漏洞，并提醒大家处理 XML 和 SAML 时要非常小心。 • kinow 分享了 BlueSky 上展示该漏洞的视频链接。 • layer8 提到这是一个解析器不匹配漏洞，并提供了一年前的相关讨论链接。 • robmccoll 建议不要使用 SAML 和 XMLDSig，因为这些协议容易出错且难以正确实现。 • GauntletWizard 认为 SAML 设计上存在不安全因素，强调了正确解析和验证签名的重要性，并批评了 AWS 的某些签名算法。 补充讨论： • malkia 提出一个实际问题，询问旧版本的 puppet 是否会受到影响，并寻求检查依赖项的网站。 争议焦点： • SAML 和 XMLDSig 的复杂性和安全性是主要争议点。一些人认为这些协议设计上有固有的不安全性，建议转向 OAuth 等更安全的协议。

文章摘要

本文讲述了作者从一块旧硬盘中恢复了苹果1994年隐藏的恢复分区功能的经历。这一发现源于Pierre Dandumont的博客，他提到Macintosh Performa 550有一个隐藏分区，用于在系统故障时恢复软件。作者拥有的Performa 550是其童年时的电脑，因此对此特别感兴趣。经过多次尝试，作者帮助Pierre在模拟器中测试恢复CD，最终确认恢复了这个隐藏分区。尽管最初未找到，但在使用Apple HD SC Setup格式化硬盘后，发现了2,560 KB的恢复分区。该文还提到，如今使用MAME模拟器可以更容易地实现这一功能。

评论摘要
主要讨论点：1994年个人电脑在操作系统损坏后的可启动能力 不同观点： • **DOS/Windows系统的可修复性较强**：userbinator认为，在1994年的DOS/Windows环境下，即使操作系统被破坏，电脑仍然可以通过插入事先制作的启动软盘来启动。DOS系统由于其简单性，只需复制几个核心文件（如两个内核文件和一个壳文件）即可制作启动盘，因此在那个时代可以很容易地使所有软盘成为启动盘。这表明DOS系统的可修复性和灵活性较高。 补充讨论： • **启动软盘的作用**：启动软盘在DOS/Windows系统中起到了重要的修复工具作用，能够在操作系统损坏时帮助用户恢复系统。这显示了当时用户依赖启动软盘进行系统维护和修复的普遍做法。 • **系统的简单性**：DOS系统的简单性使得用户可以轻松复制和分发启动文件，进一步增强了系统的容错能力和可维护性。这种简单性在当时的个人电脑中是一个显著的优势。 争议焦点： • 虽然评论中没有直接的争议，但隐含的讨论点在于不同操作系统在面对系统损坏时的恢复能力和复杂性。DOS系统因其简单性而具有较高的可修复性，而现代操作系统在提供更多功能的同时，可能在系统损坏后的修复方面变得更加复杂。
