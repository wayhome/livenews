Gooey rubber that's slowly ruining old hard drives

作者: zdw | 评分: 89 | 评论数: 10 | 发布时间: 2025-03-02 22:11
文章摘要

本文讨论了1990年代苹果品牌的SCSI硬盘（如Quantum和Conner生产）因橡胶缓冲垫老化导致的故障问题。这些硬盘常见问题是无法正常启动，主轴转动但磁头不动，随后硬盘停止运行。作者通过拆解发现，硬盘磁头被一种逐渐变粘的橡胶垫卡住，特别是停车位置的缓冲垫问题最严重。这种橡胶垫随着时间推移分解成粘性物质，导致磁头无法移动。作者提到，通过手动移动磁头可以暂时解决问题，并使用ZuluSCSI等工具保存数据。文章还澄清了打开硬盘不会立即损坏的误解，特别是老式硬盘对环境灰尘的耐受性较强。

评论摘要
主要讨论点：硬件产品的寿命与老化问题，特别是硬盘等存储设备的耐久性及其技术挑战。 不同观点： • **硬件老化的技术难度**：[klysm] 认为，产品随着时间推移而损坏，并不完全是厂商故意设计的计划报废，而是因为实际设计中要做到耐用非常困难。 • **具体产品故障经验**：[acomjean] 分享了1990年代购买的Quantum硬盘在短时间内出现故障的经历，并提到DVD和MiniDisc等其他存储介质的寿命问题，还引用了“sticky-shed syndrome”（粘性脱落综合症）作为磁带老化的一个例子。 • **橡胶老化问题**：[ChrisMarshallNY] 指出橡胶材料的老化问题，特别是鼠标等设备的外层橡胶变得粘腻，最终不得不丢弃。 • **硬盘修复经验**：[wileydragonfly] 提到通过打开硬盘手动移动机械臂来临时修复硬盘，从而成功恢复数据，但硬盘只持续了20分钟。 • **硬盘开放操作的可行性**：[userbinator] 认为老式硬盘在短时间内打开操作进行数据恢复是可行的，特别是在干净的环境下，但长期运行则不建议。 • **半导体元件老化问题**：[doctor_radium] 提到1990年代硬盘中半导体元件（特别是电容）的老化问题，尤其是对于像Amiga这样的老式设备，救回这些设备可能面临的电子元件故障问题。 • **材料选择与耐久性**：[fuzzfactor] 讨论了硬盘或其他设备中使用的聚氨酯和低密度聚乙烯等材料的优劣，指出低密度聚乙烯在耐用性上更好，但在防震性能上不如聚氨酯。 • **老式硬盘的持久性**：[Apreche] 分享了自己仍然能正常工作的20MB外部SCSI硬盘，表明某些老式硬盘在长时间后仍然可以正常使用。 • **现代硬盘的老化问题**：[lofaszvanitt] 提出了对现代硬盘在长期不使用后突然失效的疑问，特别是Xbox360硬盘和Seagate硬盘，质疑是什么导致了这些“现代”硬盘的失效。 补充讨论： • 讨论中提到了多种存储介质的寿命问题，如MiniDiscs、磁带、DVD等，比较了不同技术在长时间后的表现。 • 有人提到具体的技术问题如“sticky-shed syndrome”和“thermal asperity”（热突起），这些是影响硬盘和磁带寿命的重要技术因素。 • 讨论中还提到了不同用户在修复和维护老式存储设备时的实际操作经验和方法。 争议焦点：争议的核心在于硬件老化是否是厂商故意设计的计划报废策略，还是由于技术和材料限制导致的不可避免的问题。

Smallpond – A lightweight data processing framework built on DuckDB and 3FS

作者: overflowcat | 评分: 179 | 评论数: 9 | 发布时间: 2025-02-28 01:56
文章摘要

Smallpond是一个轻量级数据处理框架，基于DuckDB和3FS构建。其特点包括高性能数据处理、可扩展至PB级数据集、以及无需长期运行服务的简易操作。支持Python 3.8到3.12，通过`pip install smallpond`安装。使用示例展示了数据下载、初始化会话、读取和处理Parquet文件以及保存和显示结果的步骤。文档提供了详细的指南和API参考。性能评估通过GraySort基准测试，在50个计算节点和25个存储节点的集群上，30分钟内排序了110.5TiB数据，吞吐量达3.66TiB/分钟。该项目采用MIT许可证，由Runji Wang和Yiyuan Liu主要贡献。

主要讨论点：专门化查询引擎及其在数据处理中的应用 不同观点： • [jamesblonde] 认为该查询引擎专门用于训练管道，尤其注重批量提供训练数据，具备随机读取、Arrow支持和高效检查点等特性。它利用Ray进行并行化处理，并非通用查询引擎。 • [orlp] 指出在GraySort基准测试中，默认使用Polars而非DuckDB进行排序，对此表示不解，并提供了代码链接以支持其观点。 • [dang] 提到了相关的讨论线程和其他文章，指出DuckDB的分布式能力与DeepSeek的smallpond在大数据处理方面的竞争。 • [HackerThemAll] 对DuckDB结合SQLite和/或PostgreSQL的使用表示赞赏，并对DeepSeek的工作表示感谢。 • [rubenvanwyk] 表达了对数据工程内容在Hacker News上热度的喜悦。 • [fastasucan] 提问该查询引擎的具体功能及其相对于DuckDB和Polars的优势。 • [RyanHamilton] 推荐使用其开发的QStudio作为DuckDB的SQL客户端，并提供了链接。 • [shipp02] 对DeepSeek模型的代码编写能力表示惊讶，甚至调侃如果属实可能会放弃软件工程职业。 • [lvl155] 期待未来几年能够完全抽象掉所有后端技术。 补充讨论： • 对专门化查询引擎的具体功能和应用场景存在一定的好奇和疑问，尤其是相对于其他数据库工具的优势。 • 对Polars和DuckDB在不同场景下的选择和使用存在争议，特别是默认排序工具的选择。 • 对未来技术发展趋势的期待，尤其是对后端技术抽象化的展望。 争议焦点： • 默认排序工具的选择：[orlp] 对为何在GraySort基准测试中使用Polars而非DuckDB表示不解，这是评论中主要的争议点。

CSS Custom Functions are coming

作者: Fudgel | 评分: 49 | 评论数: 20 | 发布时间: 2025-03-02 22:52
文章摘要
