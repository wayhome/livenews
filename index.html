Extensible WASM Applications with Go

作者: ingve | 评分: 103 | 评论数: 8 | 发布时间: 2025-02-14 07:08
文章摘要

Go 1.24 增强了其 WebAssembly (Wasm) 功能，新增了 `go:wasmexport` 指令和构建 WASI reactor 的能力。`go:wasmexport` 允许导出 Go 函数以供 Wasm 主机调用，促进与 Wasm 环境的更好集成。通过该指令，开发者可以轻松导出函数，如示例中的 `add` 函数，使其可在主机环境中调用。 此外，Go 1.24 支持构建 WASI reactor，这种 Wasm 模块可在初始化后多次调用，适合事件驱动应用。使用 `-buildmode=c-shared` 标志可以构建此类 reactor，并通过调用 `_initialize` 函数进行初始化。 Go 1.24 还放宽了 `go:wasmimport` 函数支持的参数类型限制，支持 bool、string、指针等类型，使 Wasm 应用开发更加自然高效。这些新特性使 Go 成为构建可扩展 Wasm 应用的强大工具。

主要讨论点：Go 语言在 WebAssembly (WASM) 上的应用及其相关问题 不同观点： • **WASM 二进制文件过大及编译速度问题**：stpedgwdgfhgdd 提到，Go 生成的 WASM 二进制文件体积很大，虽然 TinyGo 能减小体积但编译速度很慢，且需要谨慎选择导入的库。此外，由于二进制文件过大，在 Cloudflare workers 上使用 WASM 需要购买订阅服务，否则难以运行超出基础 "hello-world" 的程序。 • **函数导出方式的讨论**：donatj 认为，Go 的 WASM 导出机制没有充分利用 Go 语言本身的特性，比如可以自动导出大写字母开头的函数，而不用额外使用编译器指令。他觉得这种方式有点不符合 Go 的语言习惯，但也可以理解，因为它借鉴了 cgo 的导出方式。 • **志愿者开发与进度问题**：boomskats 指出，Go 的 WASM 功能是由志愿者开发的，因此进度依赖于他们的时间安排，不能指望 Go 官方团队来推动。 • **对低层语言的需求**：Blackarea 暗示目前可能缺少一种具有强类型和优秀 WASM 支持的低层语言，侧面反映了对当前解决方案的不满。 • **WASM 组件模型的缺失**：euroderf 提到，没有讨论到 WASM 组件模型，暗示这是一个需要关注的问题。 • **垃圾回收机制的疑问**：amelius 对 Go 和 WASM 结合时的垃圾回收机制如何运作表示疑问，这是一个技术细节上的关注点。 • **对 WASM 功能快速增加的担忧**：thrance 担心对 WASM 功能的过度追求可能会损害 Go 的年轻生态系统，认为许多功能可以通过组件模型提案来实现，而目前这种方式可能会导致需要长期支持非标准特性，如 WASI。 • **WASM 模块调试问题**：Jyaif 提出了如何调试运行在宿主程序中的 WASM 模块的问题，这是一个实际应用中的技术挑战。 补充讨论： • 评论中涉及多个技术细节问题，如 WASM 二进制大小、编译速度、函数导出方式、垃圾回收机制、组件模型支持以及调试问题，这些都是在 Go 和 WASM 结合应用中需要解决的重要问题。 • 评论也反映了社区对 Go 的 WASM 功能开发进度和方向的关注，特别是由志愿者推动的开发模式可能导致的进度延迟问题。 • 最后，评论中对 WASM 组件模型的讨论显示出对未来标准化的期待以及对现有非标准特性支持的担忧。

Linux kernel cgroups writeback high CPU troubleshooting

作者: mesto1 | 评分: 70 | 评论数: 3 | 发布时间: 2025-02-14 08:30
文章摘要

本文描述了如何利用网络数据包捕获和BPF工具调试Web服务器性能问题，最终发现了一个Linux内核性能问题。在将操作系统从CentOS升级到Ubuntu后，Apache服务器出现了 listen overflow 错误和请求延迟。最初怀疑是网络问题，但排除了这个可能性。进一步调查发现，系统CPU使用率在主机启动后几分钟内急剧上升，尤其是在Ubuntu主机上。通过分析，发现问题出在内核函数 `inode_switch_wbs_work_fn` 上，该函数占用了大量CPU资源。这个CPU问题导致网络操作看似变慢，实际上是由于高CPU占用率使得进程调度延迟。最终，通过延迟主机入池时间，避免了CPU峰值期间的性能问题。CentOS未受影响是因为其内核版本较旧。

评论摘要
主要讨论点：Linux内核中的脏页写回机制及其对系统性能的影响，特别是在高内存和IO密集型工作负载下的调优问题，以及与Kubernetes和不同文件系统配置的关系。 不同观点： • **boomskats**：认为在现代拥有大内存的服务器环境中，内核的默认脏页写回参数（如`vm.dirty_ratio`和`vm.dirty_background_ratio`）不合适，尤其是在启动新服务器时。这些参数会导致内核在大量写操作时无法及时将脏页写回磁盘，从而引发性能问题。他建议调整这些参数以优化性能，并指出该问题在Kubernetes（特别是cgroups v1）中也会导致Pod因内存问题被无故杀死（OOM）。他认为cgroups v2解决了这一问题。此外，他还提到SSD的`discard`（trim）标志可能引发性能问题，建议通过取消该标志或手动执行`fstrim`来优化性能。 • **steelbrain**：对boomskats深入分析从PHP应用层到Linux内核层的能力表示钦佩，但没有提出技术性反驳或补充。 • **manuc66**：简单说明其调查是与同事合作完成的，未提出技术性观点或反驳。 补充讨论： • **脏页写回机制的调优**：boomskats详细描述了调整内核参数（如`vm.dirty_ratio`和`vm.dirty_background_ratio`）的重要性，特别是在高内存和高IO工作负载下。 • **cgroups版本差异**：boomskats指出cgroups v1和v2在处理内存和IO操作上的不同表现，并认为cgroups v2解决了v1中的内存计算问题。 • **SSD配置问题**：boomskats提到SSD的`discard`标志可能导致性能问题，并提供了解决方案（取消`discard`标志或使用`discard=async`，并手动执行`fstrim`）。 争议焦点： • **内核默认参数是否适合现代服务器环境**：boomskats认为默认参数不合适，需要手动调优，而其他人未对此提出异议或支持。 • **cgroups版本差异的影响**：boomskats详细描述了cgroups v1和v2在处理内存和IO操作上的不同表现，并认为v2解决了v1中的问题，但未见反驳或支持意见。

Does X cause Y? An in-depth evidence review

作者: l0b0 | 评分: 114 | 评论数: 15 | 发布时间: 2025-02-14 06:14
文章摘要

这篇文章讨论了一个关于X是否导致Y的理论，并深入分析了相关学术文献。作者发现，尽管有数百项研究探讨这个问题，但大多数只是简单的观察性研究，仅表明拥有更多X的人群或国家也拥有更多Y。这类研究常存在样本量小、时间短或结果测量不准确的问题，且无法排除潜在的混杂因素（如同时影响X和Y的第三个因素Z）。这些观察性研究常使用回归分析来“控制”混杂因素，但这种方法存在诸多假设和潜在错误。 在排除大量不可靠研究后，作者找到1到5项更有趣且严谨的研究。这些研究利用历史事件（如古巴巴移民潮或意大利大规模赦免囚犯）进行分析，但即便如此，深入研究后仍发现结果复杂且常常相互矛盾。最终，作者提出，我们要么接受一个过于复杂的X与Y关系理论，要么回归常识猜测，因为现有研究无法提供明确答案。

文章摘要

文章讨论了AI模型在开发者工作流中的整合如何可能抑制新技术的采用。由于AI模型的训练数据截止日期和系统提示的影响，模型对某些技术存在偏好，导致开发者在选择技术时倾向于那些AI工具支持较好的选项。这种偏好放大了文档和支持可用性在决策中的重要性，但同时也造成了“知识差距”——即AI模型的知识更新滞后，无法有效支持最新技术或功能。 这种知识差距不仅影响新框架的采用，还影响现有技术的新特性。由于模型无法提供最新信息，开发者可能不愿采用这些新技术，从而减少相关内容的产生，形成负面反馈循环。此外，AI工具常表现出对某些技术（如React和Tailwind）的偏好，即使在用户未明确要求的情况下也可能默认使用这些技术，进一步影响技术选择。 总之，AI模型的局限性和偏好可能无意中抑制了技术创新和多样性。
