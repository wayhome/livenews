The FFT Strikes Back: An Efficient Alternative to Self-Attention

作者: iNic | 评分: 125 | 评论数: 16 | 发布时间: 2025-02-26 09:57
文章摘要

文章介绍了一种名为FFTNet的模型，作为自注意力机制的有效替代方案。FFTNet利用快速傅里叶变换（FFT）实现全局token混合，时间复杂度为$\mathcal{O}(n\log n)$，相比传统自注意力的二次复杂度更高效。FFTNet通过将输入转换到频域，利用帕塞瓦尔定理保证的正交性和能量保存特性，捕捉长距离依赖关系。模型中可学习的频谱滤波器和modReLU激活函数动态强调显著频域成分，实验证明其在长序列和ImageNet数据集上的性能优于固定傅里叶模型和标准自注意力模型。

评论摘要
主要讨论点：基于傅里叶变换和卷积定理在神经网络中的应用，及其相对于传统自注意力机制的优劣比较。 不同观点： • **[xeonmc]**: 支持傅里叶变换的应用，认为通过卷积定理可以将卷积操作转化为频域中的乘法，从而简化计算，提高效率。 • **[yagizdegirmenci]**: 提到Google在2022年通过FNet引入了这一思想，但在大多数情况下，TPU上的矩阵乘法比FFT更快，暗示实际应用中需要权衡。 • **[yorwba]**: 质疑该方法如何实现因果掩码（causal masking）以及位置编码（positional embeddings），认为与非因果无位置编码（NoPE）的自注意力机制比较可能不公平，结果可能并不如宣传的那么显著。 • **[pointlessone]**: 表示对数学原理不理解，询问该方法如何等效于注意力机制，以及如何处理标记之间的位置关系。 • **[DrNosferatu]**: 赞赏该方法不仅提升了速度，还能够处理超长上下文（如超过16M的标记），对未来应用充满期待。 • **[farhanhubble]**: 提出可以将神经网络视为黑盒信号处理单元，并思考可以从信号处理中借用哪些其他技术。 • **[avereveard]**: 提到闪注意力（flash-attention）已经是n log n的复杂度，暗示可能不需要新的方法。 • **[ipunchghosts]**: 赞赏提出该想法的人，表现出对该想法的认可。 • **[TheDudeMan]**: 幽默地表示机器将在频域中思考，暗示该方法可能带来深远影响。 • **[A7C3D5]**: 幽默地将FFT理解为最终幻想战略版（Final Fantasy Tactics），与技术讨论无关。 • **[larodi]**: 表示这些数学讨论越来越复杂，暗示对该话题的理解困难。 • **[cs702]**: 简要总结了FNet的核心思想和步骤，帮助理解讨论。 补充讨论： • 争议焦点在于傅里叶变换方法如何处理因果掩码和位置编码，以及与传统自注意力机制比较的公平性。 • 时间复杂度的比较也是一个讨论热点，特别是传统自注意力、闪注意力和新型方法的复杂度。 • 有人提出可以将信号处理技术更多地应用于神经网络，拓展了讨论范围。

Telescope – an open-source web-based log viewer for logs stored in ClickHouse

作者: r0b3r4 | 评分: 84 | 评论数: 11 | 发布时间: 2025-02-26 08:28
文章摘要

Telescope是一款基于网页的日志查看工具，旨在为存储在ClickHouse中的日志数据提供直观的浏览和分析界面。用户可以配置与ClickHouse数据库的连接，通过查询、过滤和可视化工具高效分析日志数据。它支持多源管理、动态数据可视化、基于角色的访问控制（RBAC）和GitHub认证等功能。目前处于βeta阶段，未来计划增加更多功能，如SQL查询支持、快照存储和实时日志追踪等。用户可以通过Docker本地运行或访问在线演示实例进行体验。项目采用MIT许可证，代码主要使用Vue.js、Django和Python开发。

DeepGEMM: clean and efficient FP8 GEMM kernels with fine-grained scaling

作者: mfiguiere | 评分: 334 | 评论数: 14 | 发布时间: 2025-02-26 01:02
文章摘要

DeepGEMM是一个用于FP8矩阵乘法的高效库，专注于清洁代码和精细缩放，支持普通和专家混合（MoE）分组GEMM。该库使用CUDA编写，安装时无需编译，通过轻量级的即时（JIT）模块在运行时编译内核。它专为NVIDIA Hopper张量核心设计，采用CUDA核心两级累加来解决FP8精度问题。库的核心由约300行代码组成，简洁易学，性能可媲美甚至超过专家调优的库。 性能测试显示，在不同矩阵形状下，DeepGEMM的加速比在1.0x至2.7x之间。该库需要Hopper架构GPU、Python 3.8及以上、CUDA 12.3及以上，推荐使用CUDA 12.8以获得最佳性能。安装后可通过Python项目导入并使用。 DeepGEMM仅包含GEMM内核，要求LHS缩放因子为TMA对齐并转置，且仅支持NT格式。提供简单的PyTorch实用函数，但主要优化集中在GEMM内核。

评论摘要
主要讨论点：围绕FFMA SASS interleaving技术优化及其相关影响的讨论 不同观点： • Bimos分享了在CUTLASS FP8内核中通过修改编译后的二进制文件中的FADD指令和FFMA指令，成功提升了性能，特别是在FP8 GEMMs的细粒度扩展上创造了更多重叠指令的机会。这种优化是通过翻转指令中的特定位（yield位和reuse位）实现的，并取得了显著的性能提升（10%以上）。 • Fulafel对编译器技术是否能从高层代码中提取出如此高性能表示疑问，并提出传统编译器技术或AI辅助优化代理是否能实现类似的优化。这表明对编译器技术进步的期待以及当前技术局限性的认识。 • Jmward01对低精度优化的长期有效性提出质疑，认为当前模型的稀疏性可能是因为训练方法不佳，而非模型本身应该如此。这暗示了对当前优化方向的潜在担忧。 • WiSaGaN认为这些开源项目主要服务于大型企业，而不是一般想要学习和在消费硬件上运行模型的人，反映了开源软件在实际应用中的利益导向。 • Alecco表达了对开放源合作模式被大型企业接受的希望，特别是MIT许可证下的项目。 • Nbonaparte指出Blackwell（MXFP）的原生微缩支持可能会使这种手动优化变得无意义，因为Blackwell提供了更细粒度的原生支持。 • Niemandhier对未公开指令的存在表示不解，认为即使是未完全可靠的指令也应提供给用户，质疑安全通过 obscurity 的有效性，并指出竞争对手的逆向工程能力。 • Dr_kretyn虽然表示对该技术细节不了解，但赞赏这种分享和改进，认为这对整个社区有益。 • Buyucu对deepseek团队的努力表示高度赞赏。 • M3kw9以戏谑的方式询问这种技术优化的实际应用价值，反映了部分读者对实际应用场景的关注。 • Cde-v提到该讨论与NVDA即将发布的结果时机上的巧合，暗示了对市场动态的关注。 补充讨论： • Fulafel进一步提到相关技术主要基于Python实现，并提供了链接，指向了具体的开源项目DeepGEMM，提供了进一步了解和研究的路径。 争议焦点： • 低精度优化的长期有效性和合理性存在争议，部分观点认为当前模型的稀疏性可能源于不良的训练方法，而非模型本质需求。 • 未公开指令的存在和披露问题，部分观点认为即使不完全可靠也应开放给用户，以促进技术进步和应用。

Iterated Log Coding

文章摘要

Adam Scherlis提出了一个新的计算机实数编码格式——迭代对数编码（Iterated log coding）。该格式通过一系列符号位来表示实数，从数的正负开始，逐步对数值取对数并记录其符号，从而实现对极大或极小数值的精确表示。每个符号位将可能值的范围进一步缩小，最终通过足够多的符号位可以达到任意精度。 该格式具有以下特点： 1. 使用多个符号位表示数值，并通过Gray码和字典序排列实现编码。 2. 通过在符号转换为比特时附加一个额外的"1"和任意数量的零进行填充，以确保每个比特串表示一个确定的值。 3. 所有零的比特串表示NaN。 该格式展示了以下优点： - 字典序排列 - 数值范围广泛 - 对称于零 - 任何值都可表示 该实现展示了7位编码的特性，足以展示该格式的核心功能。

Understanding Surrogate Pairs: Why Some Windows Filenames Can't Be Read

作者: feldrim | 评分: 61 | 评论数: 11 | 发布时间: 2025-02-24 12:19
文章摘要

本文讨论了Windows系统中出现的一种怪异但无害的行为，即任务管理器中出现带有方块符号的执行文件名。这些文件名由于使用UTF-16编码中的替代对（surrogate pairs）而无法正确显示。尽管文件名看似可疑，但实际上只是简单的“Hello World”应用程序，且没有安全风险。Windows使用UTF-16编码处理文件名，允许使用替代对来表示Unicode中的扩展字符。然而，当字符处理不当导致孤立或畸形的替代对时，就会出现无法读取的文件名。即使操作系统能正确执行这些文件，但其显示可能出现问题。文章还介绍了如何手动创建这些替代对，并提供了Python代码示例来生成不可渲染的文件名。

评论摘要
主要讨论点：文件名编码、操作系统对文件名的处理及其问题 不同观点： • [account42] 认为Windows文档中关于文件名使用UTF-16编码的说法不准确，实际上Windows使用的是WTF-16，这种编码可能会包含不合规的UTF-16字符。 • [dwdz] 提供了在Linux系统上的实际例子，展示脚本可以正常创建和列出不同文件名的文件，表明Linux对文件名的处理相对正常。 • [mofeien] 对通过UTF-8编码转换到代码点的过程感到困惑，但通过实验发现可以直接使用`chr()`函数处理代码点。 • [feldrim] 作为原帖发布者，补充了关于Windows和Linux（通过WSL）处理文件名不同的信息，特别是关于区域设置（locale）的影响，并寻求更多相关信息。 • [kzrdude] 回忆了Mac OS X某个版本升级导致Unicode文件名无法访问的问题，表示仍然好奇这是否是一个有意为之的破坏性变更。 • [Devasta] 对UTF及其他编码尝试涵盖所有字符的做法持批评态度，半开玩笑地建议全球统一使用Rotokas语言。 • [theiebrjfb] 支持Linux系统，认为Linux在处理文件名编码问题时更简单，且具备更多现代文件系统的优势，如文件系统级别的快照功能。 补充讨论： • 讨论中多次提到"代理对"（surrogate pairs）的问题，尤其是在UTF-16编码中，以及在不同编程语言（如JavaScript）中处理代理对时的潜在问题。 • 不同操作系统（Windows、Linux、Mac OS X）在文件名编码处理上的差异引发了争议，特别是Windows的WTF-16和Linux的UTF-8处理方式。 • 文件名编码问题在实际应用中可能引发实际问题，如[kzrdude]提到的Mac OS X升级导致文件无法访问的情况，以及[somewhereoutth]和[rob74]提到的JavaScript中处理字符串时的代理对问题。 争议焦点： • Windows文件API文档中关于UTF-16编码的描述是否准确，以及WTF-16的具体影响。 • 不同操作系统在处理文件名编码时的差异及其带来的实际问题。 • 对UTF等编码系统的批评与支持，特别是在全球化背景下处理多语言文件名的问题。
