文章摘要

文章摘要

IronRDP 是一个使用 Rust 实现的微软远程桌面协议（RDP）项目，注重安全性。它支持多种视频编解码器，如未压缩的原始位图、RLE 位图编解码、RDP 6.0 位图压缩和 Microsoft RemoteFX。项目包含一个基于 IronRDP 套件的完整 RDP 客户端，使用异步 I/O，并提供一个简单的截图示例展示同步阻塞方式的使用。此外，项目还提供了如何在服务器上启用 RemoteFX 的指南。IronRDP 采用 Apache-2.0 和 MIT 双重许可，拥有 595 个星标和 59 个分支，主要使用 Rust 语言开发。

主要讨论点：IronRDP（一个独立实现的RDP工具）的性能、应用场景、跨平台支持及其与其它远程桌面协议（如VNC、SPICE）的对比。 不同观点： • [kayson] 认为MS RDP是最佳的远程桌面客户端和协议，强调其出色的性能、多显示器支持以及广泛的可用性（95%的电脑上都有）。缺点仅仅是它是微软的产品。 • [graynk] 提到Iron前缀让他联想到.NET和其它相关项目（如IronLanguages和IronSoftware），可能对命名产生混淆。 • [jeroenhd] 对IronRDP在README中的演示印象深刻，并注意到其包含服务器代码。他提出Proxmox等工具是否能使用IronRDP作为VNC或SPICE的更高效替代方案，因为后两者存在速度慢或工具稀缺的问题。 • [geenat] 关心IronRDP是否实现了服务器端，还是仅支持客户端，并询问是否支持Linux。 • [nailer] 提到Fedora最近在安装程序中放弃了VNC支持，转向了RDP，可能与Wayland相关。并指出RDP在开源世界正逐渐成为主流。 • [iJohnDoe] 询问IronRDP是否支持Linux，并询问是否有Linux发行版内置RDP支持。 • [alexpadula] 对项目表示赞赏，简单直接。 • [shravankumar8] 简短表示对项目的惊叹。 • [38] 提到无法在Windows上构建IronRDP，给出了相关问题的GitHub链接，表明Windows支持存在问题。 补充讨论： • IronRDP的命名可能引发与.NET等项目的混淆。 • IronRDP在Linux及跨平台支持上的潜力被多次提及，尤其是Linux社区对RDP的需求。 • 对IronRDP能否替代VNC和SPICE进行了讨论，特别是在Proxmox等工具中的应用场景。 • Windows上的构建问题被指出，显示出在跨平台支持上仍有挑战。 争议焦点： • 主要争议在于IronRDP是否能在不同平台（尤其是Linux）上顺利运行，以及它是否真的能替代VNC和SPICE等传统远程桌面协议。

文章摘要

这篇文章回顾了1997年Interplay在圣诞节推出的两款CRPG游戏：《Fallout》和《Descent to Undermountain》。《Fallout》获得广泛好评，而基于《龙与地下城》的《Descent to Undermountain》则反响平平。1998年，Interplay计划推出《Fallout》续作和另一款《龙与地下城》游戏，但市场反应与前一年截然相反。 《Fallout 2》的开发在原版发售前就已列入计划，因公司财务困境，快速推出续作显得更为必要。尽管最初设计提案未获通过，但在《Fallout》主创团队疲惫不堪的情况下，他们仍被要求参与续作开发。最终，主创Tim Cain、Leonard Boyarsky和Jason Anderson在不满中辞职，后续开发由Feargus Urquhart及其团队接手。开发过程松散，许多员工在不同项目间流动，最终《Fallout 2》在多方努力下完成。

文章摘要

本文讨论了在代码库中创建新类型的犹豫现象，尤其是在Java和Go项目中。作者指出，这种犹豫表现为使用大量局部变量、多个函数参数或返回值，以及扩展现有类型而不是创建新类型。作者认为，这种现象可能源于对改变“大局设计”的恐惧，尤其是对新手开发者而言，他们可能觉得创建新类型是重大决策。作者批评了这种思维，强调类型系统正是为了将相关信息组织成易于使用的整体。他建议像C和Go语言文化那样，根据需要轻松创建新类型，哪怕只用于单个函数，以简化代码和逻辑。虽然过度使用新类型会增加认知负担，但适当使用可以使代码更清晰和易于维护。作者鼓励开发者勇于创建新类型，而不是过度依赖模型类型或通过函数文档说明特殊情况。

评论摘要
主要讨论点：在编程中对新类型或名称的使用及其利弊 不同观点： • **beders 的观点**：认为在Java中，每个小元素都需要一个名称，而如果没有合适的名称，可能就不需要创建新类型。他以Clojure为例，展示了如何通过解构从map中直接提取键值，而不需要为参数命名或创建新类型。他认为在Java中强制命名和创建新类型可能是不必要的，尤其是在没有合适名称的情况下。 • **salmonellaeater 的观点**：指出其工作中的代码库通常不愿意创建新类型，且这种现象难以解释。他认为像"CreateSubscriptionRequest"这样的小型专用类型在测试和维护代码时非常有价值。他提到TypeScript允许声明类型来描述一些值，而不需要实现新接口，这减少了摩擦。尽管TypeScript中声明新类型是无成本的，但在其他语言如Java中，由于需要包装新类，工程师们往往不愿意引入新类型。 • **parpfish 的观点**：认为将代码视为类型转换，而不是算法和过程，可以使代码更容易维护。通过系统地命名和组织代码结构，编写单元测试也变得更加简单。他强调了类型转换和结构化代码的优势。 • **paulddraper 的观点**：认为不应该害怕使用数据库表，可以根据需要创建大量表，而不必强制复用或进行泛型设计。他指出，数据库管理系统可以处理大量表，因此不需要过度担心表的数量。 补充讨论： • **争议焦点**：在不同编程语言中，创建新类型或使用名称的必要性和成本。特别是在Java中，由于强制命名和创建新类的摩擦，导致许多工程师不愿意引入新类型。而在TypeScript中，这种情况较少，但仍存在不愿意声明新类型的现象。 • **例子和论据**： - beders 提供了Clojure中使用解构的例子，展示了无需命名参数或创建新类型的方法。 - salmonellaeater 提到了TypeScript中声明类型的便利性，以及在Java等语言中引入新类型的额外成本。 - parpfish 分享了通过类型转换和结构化代码，使维护和测试变得更加简单的经验。 - paulddraper 强调了在数据库设计中不必过度担心表的数量，可以根据需要创建大量表。 整体来看，讨论围绕在不同编程实践中对类型和名称的使用，以及如何在代码可维护性和设计简洁性之间取得平衡。

France rejects backdoor mandate

作者: hn_acker | 评分: 540 | 评论数: 16 | 发布时间: 2025-03-21 20:35
文章摘要

法国国民议会明智地拒绝了削弱端到端加密的法案，该法案名义上旨在打击毒品贩运。尽管内政部施加了很大压力，议员们仍投票否决了一项本会强制Signal和WhatsApp等通讯平台允许秘密访问私人对话的条款。该提案实际上是监控愿望单，采用了一种被称为“幽灵”参与者的后门方法，安全专家警告这将引入系统性漏洞并破坏安全通讯平台的信任。法国议员的决策捍卫了数字权利、隐私和安全，表明加密技术支持基本人权，而非社会的敌人。这一胜利为全球其他国家树立了榜样：不应以公共安全为名牺牲基本权利。未来仍需保持警惕，抵制类似提案。

评论摘要
主要讨论点：法国及欧洲在加密通信和后门问题上的政治和技术争议 不同观点： • **政治家缺乏技术理解，导致错误决策**： - [palata] 认为政治家普遍缺乏对加密技术的理解，尤其是后门问题。他们不具备相关技术背景，却反复要求在加密通信中加入后门，这显示出他们对基本技术原理的无知。欧洲应加强隐私保护，而非削弱它。后门只会让敌对势力更容易获取私人数据。 - [buybackoff] 进一步指出，法国国民议会中的某些政客，如内政部长Retailleau，要么技术上完全不称职，要么就是在撒谎。政客们声称可以“选择性应用数学（加密）”，但这在技术上不可行。 • **政治背景和动机质疑**： - [phtrivier] 认为国民议会目前非常分裂，投票反对后门更多是一种针对政敌的政治操作，而非纯粹出于技术或隐私保护考虑。当前的反后门立场可能只是政客们借机打击对手的手段，并不代表他们在所有情境下都会反对后门。 - [thomassmith65] 则认为公众在面临安全和隐私之间的选择时，往往会支持后门，尤其是当涉及打击犯罪和恐怖主义时。E2EE（端到端加密）在未来很难在大型科技公司中持续存在，特别是在重大事件发生后，公众对加密技术的支持可能会减弱。 • **技术可行性和实际效果争议**： - [aucisson_masque] 指出，后门对打击犯罪实际上效果有限。犯罪分子可以轻易找到其他加密方法，甚至使用非数字手段加密通信。因此，后门只会影响普通用户，而非真正的犯罪分子。 - [spapas82] 强调，只有去中心化的加密方式（如PGP/GPG）才能保证真正的安全，依赖中心化的服务（如WhatsApp、Signal）进行加密是不可靠的。 • **政策趋势和未来发展**： - [nickslaughter02] 提到，尽管法国反对加密后门，但仍支持聊天控制（即设备上的通信扫描）。欧盟多数国家支持该提案，预计会通过。 - [econ] 预测未来可能会出现能精确识别可疑对话的AI技术，这可能会在保持加密的同时满足执法需求。 补充讨论： • **历史背景与法律传统**： - [AceJohnny2] 介绍了法国自1978年以来的CNIL（国家信息与自由委员会），该机构旨在防止公民隐私受到侵犯，具有历史意义。尽管如今CNIL的影响力可能减弱，但它为隐私保护奠定了法律基础。 • **法国的领导地位与乐观展望**： - [ziofill] 和 [grej] 对法国在保护隐私和捍卫自由方面的领导作用表示赞赏，并对法国的未来持乐观态度。 • **哲学反思**： - [Etheryte] 强调自由和隐私需要持续捍卫，不能一劳永逸。每个人都需要积极参与，否则权力会被滥用。 争议焦点： - **后门政策的有效性和技术可行性**：一方认为后门是打击犯罪的必要手段，另一方则认为后门不仅无效，反而会损害普通用户的隐私。 - **政治动机与公众利益的冲突**：有观点认为反后门的立场可能是政治操作，而非真正出于技术或隐私考虑。
