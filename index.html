Extensible WASM Applications with Go

作者: ingve | 评分: 44 | 评论数: 4 | 发布时间: 2025-02-14 07:08
文章摘要

Go 1.24 引入了对 WebAssembly (Wasm) 功能的增强，包括新增的 `go:wasmexport` 指令和构建 WebAssembly System Interface (WASI) reactor 的能力。这些功能使开发者能够将 Go 函数导出为 Wasm 模块，便于与 Wasm 主机集成，并扩展基于 Go 的 Wasm 应用的可能性。 `go:wasmexport` 指令允许将 Go 函数导出，供 Wasm 主机调用。例如： ```go //go:wasmexport add func add(a, b int32) int32 { return a + b } ``` 此外，Go 1.24 支持通过 `-buildmode=c-shared` 构建 WASI reactor，这种模块在初始化后可以多次调用，而不会自动终止。例如： ```sh GOOS=wasip1 GOARCH=wasm go build -buildmode=c-shared -o reactor.wasm ``` 这些功能使得 Go 开发者可以更轻松地创建可扩展的 Wasm 应用，并通过导出的函数实现动态功能扩展，而无需重新编译整个应用。Go 1.24 还放宽了 `go:wasmimport` 函数的类型限制，支持更自然的参数传递。

评论摘要
主要讨论点：Go语言与WebAssembly（WASM）相关的优缺点及其实际应用中的挑战 不同观点： • **boomskats**：对Go在WebAssembly方面的工作表示赞赏，但指出这些工作是由志愿者而非Go团队完成的，因此进展取决于志愿者的空闲时间。这表明开发进度可能较慢且不确定。 • **stpedgwdgfhgdd**：指出Go生成的WASM二进制文件体积过大，虽然TinyGo可以解决这一问题，但编译速度慢，且需要谨慎选择导入的库（特别是涉及反射的库）。此外，在Cloudflare workers上尝试Go WASM时，由于二进制文件的大小问题，可能需要购买订阅服务。这强调了在实际应用中使用Go WASM的困难和局限性。 • **euroderf**：提到没有涉及WASM组件模型的工作。这表明对WASM组件模型支持的关注和期待。 • **Jyaif**：提出如何调试运行在宿主程序中的WASM模块的问题。这反映了对WASM模块调试难度的关注和需求。 补充讨论： • 志愿者贡献与官方团队支持的差异对项目进度的影响。 • Go WASM二进制文件体积过大在实际部署中的限制，特别是对云服务如Cloudflare workers的影响。 • 对WASM组件模型的支持和调试工具的关注和需求。 争议焦点：Go WASM在实际应用中的可行性和易用性，特别是二进制文件体积和编译速度的问题，以及缺乏对一些重要功能（如WASM组件模型和调试工具）的支持。

Linux kernel cgroups writeback high CPU troubleshooting

作者: mesto1 | 评分: 11 | 评论数: 0 | 发布时间: 2025-02-14 08:30
文章摘要

本文描述了一次针对网络服务器性能问题的调查，最终发现是Linux内核中的性能问题。在将操作系统从CentOS升级到Ubuntu后，Apache web服务器出现了 listen overflow 错误和请求延迟。初始怀疑是网络问题，但通过网络包捕获和BPF工具分析排除了网络因素。进一步调查发现，Ubuntu主机在启动后的几分钟内系统CPU使用率异常升高，导致请求处理变慢。问题源于内核中的 `inode_switch_wbs_work_fn` 函数，该问题仅出现在较新的Ubuntu内核中，CentOS未受影响。最终通过延迟主机池化时间避免CPU高峰，解决了该问题。

NAT Is the Enemy of Low Power Devices

作者: hasheddan | 评分: 52 | 评论数: 10 | 发布时间: 2025-02-10 19:12
文章摘要

文章讨论了网络地址转换（NAT）对低功耗设备通信的影响。NAT解决了IPv4地址不足的问题，通过让多个设备共享一个公共IP地址，但在设备需要接收数据或进行点对点通信时产生挑战。NAT表只在有外出通信时维持映射，若一段时间无通信，映射会被删除，导致后续 incoming 数据被丢弃。这对电池有限、带宽紧张的设备尤其不利，因为需要频繁发送“保持连接”消息以维持映射。此外，重新连接时可能分配不同地址，导致需重新建立会话，增加通信开销。最糟的情况是设备需定期发送数据但频率不足以维持NAT映射，如每30分钟发送一次传感器读数，而NAT超时为5分钟，设备需频繁发送保持连接消息。

评论摘要
主要讨论点：物联网（IoT）设备的安全性、连接性及网络架构问题，特别是与NAT、会话管理和设备软件质量相关的技术讨论。 不同观点： • **procaryote** 认为IoT设备软件质量参差不齐且很少更新，因此建议将IoT设备放在单独的WiFi网络上，仅允许它们与可更新的本地控制器通信，并假设这些设备可能在出厂时已被入侵，尤其是闭源设备。 • **vv_** 强调了IoT设备在电池供电或数据预算有限的情况下，会话管理的问题，尤其是与服务器的重新连接和加密操作带来的开销。他们建议使用像Zephyr RTOS这样更成熟的软件平台来缓解这些问题，但指出许多公司为了节省硬件成本而忽略了软件工程的高昂代价。 • **dent9876543** 认为NAT实际上是一种保护机制，防止IoT设备主动向外网发送数据。他们不认为NAT是问题，反而视其为一种对抗设备不必要外联的工具。 • **kazinator** 提出了端口转发作为解决方案，允许设备通过特定端口与外部通信，类似于在家庭网络中运行服务器的方式。 • **justahuman74** 指出IPv6没有被提及，暗示这可能是一个潜在的解决方案。 • **wavesound** 补充说现代防火墙/路由器中的状态检测（SPI）也有超时问题，不仅仅是NAT。 • **philjohn** 提到了Thread协议，认为它可以通过边界路由器解决内部网络与外网通信的问题，特别适合低功耗设备。 • **gabesullice** 对原文中"less important"的措辞提出质疑，认为在某些低功耗场景中，立即推送数据的能力应该是更重要的。 • **arcmechanica** 肯定了NAT的历史贡献，认为它对互联网的扩展至关重要。 • **hermanubis** 提出了使用服务器发送令牌给客户端的简单解决方案来维持会话。 补充讨论： • **争议焦点**：NAT是否为问题所在，以及如何有效解决IoT设备与服务器的会话管理问题。部分评论者认为NAT是一种保护手段，而其他人则认为它带来了连接性问题。 • **技术方案**：包括端口转发、Session IDs、Session Tickets、IPv6和Thread协议等技术方案的讨论，反映了不同的技术偏好和实际应用中的权衡。 • **软件质量问题**：vv_指出了硬件成本与软件质量之间的矛盾，强调了使用成熟软件平台的重要性。
