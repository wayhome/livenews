Peer-to-peer file transfers in the browser

作者: keepamovin | 评分: 481 | 评论数: 43 | 发布时间: 2025-03-12 12:08
文章摘要

FilePizza 是一个基于浏览器的对等（P2P）文件传输工具，利用 WebRTC 技术实现文件共享，无需将数据存储在中间服务器，确保传输快速、私密且安全。其主要特点包括：全新支持暗模式的用户界面，适用于大多数移动浏览器；文件直接从上传者浏览器传送到下载者浏览器，速度更快；支持上传者监控传输进度并随时中止；增强的安全措施如密码保护和报告功能；支持多文件上传并以zip格式下载；以及使用 Service Worker 实现流式下载等。FilePizza 使用 Next.js、Tailwind、TypeScript、React 和 PeerJS 等技术开发，并可选择使用 Redis 存储服务器状态。项目遵循 BSD 3-Clause 许可证。

评论摘要
主要讨论点：P2P文件传输工具的现状、优缺点、技术实现及未来发展 不同观点： • **smusamashah**：列举了多个浏览器和命令行界面的P2P文件传输工具，提到LimeWire收购了多个优秀工具，如ShareDrop和SnapDrop，目前pairdrop.net是剩下的少数选择之一。 • **Springtime**：回顾了Opera浏览器在2010年左右推出的Unite功能，包括P2P文件传输、冰箱式的GUI等，强调了Opera当时集成了多种功能但仍然保持了精简。 • **latexr**：指出大多数P2P传输工具生成的复杂URL不便于电话沟通中分享，推荐了file.pizza和pairdrop.net，但仍期待更简便的CLI+浏览器结合的解决方案。 • **modeless**：希望实现本地端点通过WebRTC进行发现，而不需要互联网连接或特殊URL，目前无法做到离线本地分享。 • **mary-ext**：担心PairDrop和SnapDrop被LimeWire收购后的发展方向，暗示对LimeWire的加密货币和AI业务方向存疑。 • **crazygringo**：质疑WebRTC在需要TURN服务器时的隐私和安全问题，如何确保数据不被TURN服务器读取，是否使用了端到端加密（E2EE）。 • **amelius**：感叹P2P传输问题早在几十年前就应该解决，但一直缺乏稳健且非商业化的解决方案，并询问IPFS项目的现状。 • **DeathArrow**：建议混合系统，类似于Dropbox，在没有其他对等点或速度慢时使用服务器加速。 • **arthursw**：推荐PairDrop，并提供了其GitHub链接。 • **Rygian**：提到Magic Wormhole作为一个非浏览器内置的P2P传输工具。 • **cassepipe**：推荐使用torrent客户端如privtracker，认为其可以后台运行，不依赖浏览器标签页。 • **0xbadcafebee**：质疑是否真的需要复杂的设置来实现浏览器间的WebRTC连接，认为简单的HTML和JavaScript应该足够。 • **ebfe1**：分享了自己受Firefox Send启发制作的基于WebRTC的文件传输工具，并提供了其AWS部署链接。 • **martylamb**：提到自己早年制作的基于Java Applet的文件传输工具NearShare，当时还没有WebRTC技术。 • **janandonly**：尝试了多个WebRTC文件传输工具，但效果不理想，目前测试Keet并对其抱有期待。 补充讨论： • WebRTC技术的隐私与安全问题：特别是TURN服务器的中继传输是否会泄露数据，E2EE是否应用。 • 对LimeWire收购多个P2P工具的担忧，可能影响这些工具的未来发展方向。 • 用户对简化URL的需求，以及对CLI与浏览器结合的P2P传输方案的期待。 • 本地端点发现和离线分享的技术挑战。 • 混合系统的建议，结合P2P和服务器加速的优势。 争议焦点： • WebRTC在TURN服务器传输时的隐私与安全保障问题，是否需要信任TURN服务器或使用E2EE。 • 对LimeWire收购多个P2P工具后发展方向的担忧与不确定性。
